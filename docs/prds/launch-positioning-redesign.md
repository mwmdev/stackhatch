# StackHatch Launch Positioning and Experience Redesign

## Summary

StackHatch will launch as a visual architecture workspace for software developers who want to
understand and keep sight of how a codebase fits together. The primary workflow is deliberately
narrow: give StackHatch a public GitHub repository, inspect the architecture map, ask questions in
the context of that map, compare credible alternatives, and re-scan when the project changes.

The public site and signed-in experience will be redesigned around that workflow. The homepage's
signature will be a real, read-only map of the public StackHatch repository generated by
StackHatch itself. Visitors can explore that proof without signing in; GitHub authentication and
Anthropic BYOK are required only when they map their own repository.

This replaces the current multi-persona story for founders, agencies, devtool teams, clients, and
investors. AI remains an enabling mechanism rather than the headline. The new experience should
feel like a precise developer instrument: quiet, direct, visually grounded in system topology, and
free of generic feature-card marketing.

## Goals

- Make the product's purpose understandable within the first viewport: see how a codebase fits
  together.
- Optimize the visitor journey from a public GitHub URL to a successfully generated first map.
- Demonstrate the real product before authentication with a no-cost, no-AI-call StackHatch demo.
- Establish one coherent vocabulary and visual identity across the homepage, login, BYOK setup,
  dashboard, project creation, editor shell, support, teams, invitations, administration, and
  legal pages.
- Support launch adoption through measurable visits, activation, repeat map use, and GitHub
  interest.
- Make the free, tierless BYOK model clear before a visitor starts authentication.
- Keep every public claim within the product's actual launch capabilities.

## Non-Goals

- Private repository analysis at launch.
- Automatic repository synchronization or continuous architecture monitoring; re-scans are
  initiated by the user.
- File-level dependency graphs, call graphs, or claims of code-verification accuracy.
- Repositioning StackHatch as enterprise architecture governance, a C4 modelling suite, a general
  whiteboard, or a generic AI diagram generator.
- Adding pricing, subscriptions, feature tiers, quotas, or a server-funded AI trial.
- Adding an AI provider other than Anthropic.
- Rebuilding every editor interaction. The editor keeps its productive density while adopting the
  new shell, language, tokens, and onboarding cues.
- Leading with PRD import, blank canvases, teams, exports, investors, clients, or agencies. These
  capabilities remain available but are supporting paths.

## Audience and Positioning

### Primary audience

Individual software developers who need a useful high-level mental model of a public codebase.
The initial wedge includes:

- A developer maintaining their own project who wants the architecture to remain visible while
  the implementation evolves.
- A developer joining or taking over an existing public project who needs to identify its main
  systems and relationships quickly.
- An open-source contributor evaluating a repository before deciding where and how to contribute.
- A technical lead who wants a shared visual starting point for an architecture conversation.

These users are comfortable with GitHub and can understand an API-key setup flow. They value a
useful overview and the ability to interrogate it more than formal modelling notation.

### Secondary audience

- Small engineering teams that share and comment on maps after an individual creates one.
- Technical founders using a brief or blank canvas to explore a not-yet-built system.
- Developers using StackHatch to compare a technology choice or prepare a diagram/PRD handoff.

Secondary audiences must not dilute the homepage thesis.

### Core job to be done

> When I need to understand or keep track of a codebase, show me its main architectural pieces and
> how they relate, then let me question and challenge that model so I can make changes without
> losing sight of the whole system.

### Category and promise

- Category: visual architecture workspace for GitHub codebases.
- Primary promise: **See how your codebase fits together.**
- Supporting promise: Turn a public GitHub repository into an editable architecture map. Ask
  questions in context, compare alternatives, and re-scan as the project changes.
- Proof: StackHatch maps its own public repository on the homepage.
- Commercial model: Free to use; AI actions run with the user's Anthropic API key.

### Language rules

Use plain developer language: repository, codebase, map, system, component, connection, question,
alternative, decision, and re-scan.

Avoid:

- “AI-powered architecture design tool” as the category.
- “Before the build hardens,” “decision-ready,” “technical handoff,” “working proof,” and other
  abstract campaign phrases.
- Claims about “any repository,” private repositories, automatic synchronization, code-level
  dependency accuracy, or a map always being current.
- Treating founders, agencies, clients, investors, and engineering teams as equal audiences.
- Describing decorative screenshots as proof when a real product view is available.

### Scope and confidence

- Geography and language: global, English-language launch.
- Time window: current launch positioning as of July 2026.
- Decision supported: initial ICP, product category, homepage message, and developer-native launch
  channels.
- Overall opportunity confidence: **Medium**. Demand for codebase visualization and architecture
  understanding is visible, but StackHatch has no external usage evidence yet.
- Audience confidence: **High**. The product's working repository flow and the founder's own
  experience both point to developers seeking a visual system overview.
- Differentiation confidence: **Medium**. Questioning a visual map and exploring alternatives is a

### Demand signals and strategic implications

- GitDiagram's public repository reports roughly 15,000 GitHub stars and describes the original
  problem as understanding large open-source codebases before contributing. Its initial
  [Show HN launch](https://news.ycombinator.com/item?id=42521769) received substantial engagement,
  with recurring praise for high-level visualization and recurring concerns about accuracy,
  failure states, and large repositories. This is strong evidence for the problem and a warning
  against overclaiming fidelity.
- A separate [Hacker News discussion about codebase visualization](https://news.ycombinator.com/item?id=28074827)
  explicitly distinguishes useful semantic components and interactions from a visualized folder
  tree. StackHatch's map should describe systems, not merely files.
- Eraser now leads with automatically current codebase diagrams and onboarding. StackHatch must
  say “re-scan” rather than “always up to date” and should not imply background monitoring.
- IcePanel and Structurizr already serve formal, long-lived architecture documentation and
  governance. Competing head-on would make the product appear less capable; the opportunity is a
  faster individual-developer entry point.
- Recent research on repository graphs and visual representations suggests that structured
  repository context can improve code understanding, but it does not establish the accuracy of
  StackHatch's current inference pipeline. Treat this as category support, not product proof.

### Opportunities

- Own the transition from “show me the pieces” to “let me question the architecture.” GitDiagram
  stops near the first map; DeepWiki is conversational but documentation-led.
- Make the product demonstrate itself. A self-map is more credible and memorable than a generic
  animated illustration.
- Use the free, tierless BYOK model as a trust and control point after the value proposition, not
  as the hero's main idea.
- Launch through developer-native proof: the public repository, Show HN, relevant open-source and
  programming communities, and Product Hunt. Each destination should point to a demo that works
  without an account.

### Risks

- The current analyzer in `src/lib/github-analyzer.ts` reads repository metadata, language totals,
  and up to five root package/configuration files. It does not currently read the repository tree,
  README, application source, or verified dependencies. The intended “codebase architecture”
  promise will outrun the evidence unless analysis depth improves before launch.
- AI-generated connections may be incomplete or incorrect. The interface needs an explicit
  generated-overview label, repository/commit provenance, editable nodes, and a simple way to
  report a wrong map.
- Public-only analysis excludes many workplace onboarding cases. Every repository input must say
  “public GitHub repository”; private access belongs on the roadmap, not in launch copy.
- Requiring both GitHub login and an Anthropic key creates activation friction. The no-login demo,
  retained repository intent, and focused key setup are required mitigation.
- A broad visual redesign can damage a dense editor. Spend the visual signature on the public demo
  and keep application controls restrained.

## Information Architecture and Draft Copy

### Public navigation

Keep the header short:

- StackHatch wordmark → `/`
- Explore the demo → `/#demo` on desktop and `/demo` from compact/mobile contexts
- GitHub with a live star count when available → `https://github.com/mwmdev/stackhatch`
- Sign in → `/login?callbackUrl=/app`
- Theme control may remain as an icon, but it must not compete with the primary actions.

Do not add Features, Solutions, Resources, Pricing, Customers, or other empty SaaS-navigation
categories.

### Homepage

#### Hero

Eyebrow:

> Visual architecture for GitHub repositories

Heading:

> See how your codebase fits together.

Supporting copy:

> Paste a public GitHub repository. StackHatch maps the system, lets you ask architecture
> questions, and helps you compare other ways to build it.

Primary control:

- Repository field placeholder: `github.com/owner/repo`
- Button: **Map this repository**
- Accept a full GitHub URL or `owner/repo`; normalize and validate before continuing.
- Supporting note: **Free to use · AI features use your Anthropic API key**

Secondary action:

> Explore the StackHatch map

The field is the primary call to action. “Open the workspace,” “Get started,” and “Learn more” are
not used.

#### Self-mapping demo

Heading:

> StackHatch, mapped by StackHatch.

Supporting copy:

> This is a real, read-only map generated from the public repository. Open a component, follow a
> connection, or see how StackHatch answers a question about itself.

Show repository and snapshot provenance in compact utility text:

> `mwmdev/stackhatch · mapped from <short commit> · <date>`

The embedded view shows enough of the real map to be useful. **Explore the full map** opens `/demo`
without authentication.

#### Product sequence

This is a real ordered workflow, so four numbered steps are meaningful:

1. **Map the system.** Start from a public repository and get a visual overview of the main
   components and connections.
2. **Ask in context.** Ask how a part works or why it connects to the rest of the system.
3. **Test another direction.** Open a component and compare practical alternatives without losing
   the current map.
4. **Re-scan when the code changes.** Generate a fresh view when the repository evolves.

Use one continuous map/connection line or one real product sequence, not four independent feature
cards with repeated icons.

#### Use moments

Use three concise text rows rather than persona cards:

- **Your project:** keep the whole system visible while you work on one part of it.
- **A project you joined:** form a useful mental model before making your first change.
- **An open-source project:** understand the architecture before choosing where to contribute.

#### BYOK and trust

Heading:

> Free product. Your model. Your key.

Copy:

> StackHatch has no plans, quotas, or feature gates. Connect an Anthropic API key for repository
> analysis, questions, and alternatives. Your key is encrypted at rest and never returned to the
> browser.

Link to a concise BYOK explanation and Settings after authentication. State that Anthropic bills
model usage directly. Do not imply that StackHatch can inspect private repositories through the
user's GitHub sign-in.

#### Open-source/GitHub proof

Heading:

> Built in the open.

Copy:

> Inspect the code behind the map, report an issue, or star StackHatch on GitHub.

Actions:

- **View the source**
- **Star on GitHub** with a server-cached count when available

#### Final call to action

Heading:

> Put your codebase on the map.

Repeat the repository input rather than adding a generic button.

### Authentication and setup

The login page must preserve the value proposition and the repository the visitor intended to
map.

- Heading: **Map a repository with StackHatch.**
- Copy: **Sign in with GitHub to save the map, ask questions, and revisit it as the project
  changes.**
- Primary action: **Continue with GitHub**
- Trust note: **StackHatch analyzes public repositories only. Signing in does not grant access to
  private repositories.**

If the user arrived with a repository intent, show its normalized `owner/repo` above the action.
After authentication, return to that intent rather than the generic dashboard.

When an Anthropic key is missing, Settings becomes a focused second onboarding step:

- Heading: **Connect Anthropic to map this repository.**
- Explain exactly which actions use the key, how it is stored, and that Anthropic bills usage.
- Link to Anthropic's official key-management instructions.
- After a successful save, use **Continue to <owner/repo>** when a repository intent exists.
- Keep model selection and theme available but visually secondary during first-run setup.

### Dashboard and project creation

Make the existing repository input the dashboard's dominant action:

- Heading: **What do you want to map?**
- Repository field and **Map repository** action first.
- Existing projects follow as **Your maps**, ordered by most recently updated.
- Group PRD upload, blank canvas, and template creation under **Other ways to start**.
- Keep team creation and administration in the application navigation rather than the first-use
  content hierarchy.
- Empty state: **Map your first public repository** with a real example URL.
- Missing-key state explains the next action without blocking blank/manual work.

Project creation, team, invitation, and template pages use the same terms: map, repository,
workspace, and project. Remove inconsistent “AI assistance,” “architecture design,” and generic
welcome copy where a specific action is available.

### Architecture workspace

Do not redesign the core canvas into a marketing surface. Clarify the working vocabulary:

- **Ask about this architecture** for the chat input context.
- **Explore alternatives** for the existing alternatives action.
- **Re-scan repository** for the manual update action, with a confirmation that explains what will
  be regenerated or cleared.
- Show the source repository and latest scan time/commit when available.
- Label generated output as an architecture overview, not verified source truth.
- Add a compact **Report an incorrect map** link that pre-fills repository/project context without
  including private user content.

### Support, settings, teams, admin, and legal

- Rewrite Support around: map a repository, ask a question, explore alternatives, re-scan, BYOK,
  and known public-repository limitations.
- Keep Settings task-oriented: Anthropic connection, model, and appearance. Replace technical
  labels with explanatory labels only when precision is preserved.
- Apply the shared shell and vocabulary to team, invitation, and administration pages without
  hiding role-specific operations.
- Update Privacy to state exactly which public repository metadata/configuration is fetched, what
  is sent to Anthropic, what analytics records, and how long application data is retained.
- Keep Terms factual. Do not insert unverified security, compliance, uptime, accuracy, training,
  or data-residency claims.

## Visual Direction

### Concept

**The codebase as a navigable system map.** The design borrows from topology and technical
instruments, not terminal cosplay or enterprise blueprint clichés. Large quiet fields give the
real architecture graph room to be the identity. Lines, labels, node categories, and provenance
carry meaning; decoration does not.

The single aesthetic risk is the self-referential hero: the page opens onto StackHatch's own map,
which assembles once and then becomes an explorable product surface. Everything around it remains
restrained.

### Core tokens

- Canvas — `#F4F7FA`: cool, nearly white working surface.
- Ink — `#18212B`: primary text and strong dividers.
- Blueprint — `#2864DC`: primary action and active connections.
- Signal teal — `#168F7A`: successful paths and API/service cues.
- Decision amber — `#CE7B22`: alternatives, warnings, and decision emphasis.
- Night canvas — `#111820`: dark-mode foundation.

Existing architecture-category colors remain semantic data encodings, recalibrated for contrast
against the new light and dark canvases. They are not reused as decorative section accents.

### Typography

- Display: **Archivo**, weights 700 and 800. Its broad, engineered forms carry concise headings
  without turning the site into terminal-themed developer marketing.
- Body: **Atkinson Hyperlegible**, weights 400 and 700, retained for reading clarity.
- Utility/data: **IBM Plex Mono**, weights 400 and 600, used only for repository names, commit
  provenance, node metadata, and compact labels.

Headlines remain short and usually fit on one or two lines. Avoid uppercase eyebrow text except for
actual data/status labels.

### Layout

Use broad planes and alignment rather than nested rounded cards:

```text
┌──────────────────────────────────────────────────────────────────────┐
│ StackHatch        Explore demo       GitHub ★        Sign in        │
├──────────────────────────────────────────────────────────────────────┤
│ See how your codebase fits together.                                │
│ [ github.com/owner/repo                         ] [ Map repository ] │
│ Free to use · Bring your Anthropic key                              │
├────────────────────────── REAL STACKHATCH MAP ───────────────────────┤
│                                                                      │
│  chat / question       connected architecture graph      node detail│
│                                                                      │
│  mwmdev/stackhatch · commit · mapped date                           │
├──────────────────────────────────────────────────────────────────────┤
│ Map ───────── Ask ───────── Compare ───────── Re-scan                │
├──────────────────────────────────────────────────────────────────────┤
│ Your project        A project you joined        Open source         │
├──────────────────────────────────────────────────────────────────────┤
│ Free product / BYOK                       Built in the open / GitHub │
├──────────────────────────────────────────────────────────────────────┤
│ [ github.com/owner/repo                         ] [ Map repository ] │
└──────────────────────────────────────────────────────────────────────┘
```

- Desktop uses a maximum content width around 1200–1280 px, but the demo can bleed wider to feel
  like the actual workspace.
- Mobile stacks the input action, renders a useful static map preview first, and opens the full
  read-only demo as a dedicated viewport with node details in a bottom sheet.
- Controls use small radii (4–6 px). Large rounded feature cards, floating gradient orbs, glass
  panels, pill-heavy badges, and icon-per-sentence decoration are excluded.

### Motion

- Use one orchestrated map reveal: key nodes appear, then their connecting edges resolve. Keep it
  under roughly 700 ms and play it once.
- Do not add repeated scroll reveals, floating elements, parallax, or ambient glow.
- `prefers-reduced-motion` receives the completed static state with no transition.
- Hover and focus states communicate interactivity through line weight, color, and concise detail,
  not movement for its own sake.

## Requirements

### R1. Public self-map demo

- Add a public `/demo` route and an embeddable homepage preview using one shared read-only demo
  component.
- Generate the demo from `https://github.com/mwmdev/stackhatch` and check a sanitized architecture
  snapshot into the repository with source URL, commit SHA, and generation date.
- The full demo supports pan, zoom, keyboard navigation, node selection, connection inspection,
  and a responsive detail surface.
- Include a small set of clearly labelled preset questions and alternative views with checked-in
  answers. They demonstrate the interaction without calling Anthropic or pretending to be live
  model output.
- The public demo makes no database writes, provider requests, or authenticated API calls.
- Load a static poster as the initial/LCP state and lazy-hydrate the interactive canvas after the
  user requests it or it approaches the viewport.
- Every demo state includes a **Map your repository** action.

### R2. Repository-intent activation

- Accept full public GitHub repository URLs and `owner/repo` shorthand on the homepage.
- Validate and normalize input before starting authentication.
- Preserve the normalized repository intent through GitHub OAuth, Anthropic-key setup, and return
  navigation.
- Public repository URLs may appear in application state, but analytics events must never include
  the URL, owner, repository name, or query string.
- After prerequisites are satisfied, return the user to a prefilled confirmation action. Do not
  spend their Anthropic balance without an explicit **Map repository** action.
- Invalid, missing, private, rate-limited, oversized, and provider-failure states explain the
  problem and the next action in plain language.

### R3. Consistent product shell and copy

- Apply the new font roles, color tokens, navigation treatment, focus states, control geometry,
  and vocabulary across every route.
- Structurally redesign the homepage, login, focused BYOK setup, dashboard, new-project entry,
  support page, and public demo.
- Apply lighter shell/copy alignment to the editor, settings outside onboarding, teams,
  invitations, templates, administration, privacy, and terms.
- Preserve role checks, project/team permissions, error paths, and existing editor capabilities.

### R4. Public repository readiness and GitHub advocacy

- The `mwmdev/stackhatch` remote is not publicly accessible at specification time. Public launch is
  blocked until an unauthenticated visitor can open the repository.
- Before changing visibility, audit the current tree and retained git history for credentials,
  private endpoints, personal data, proprietary assets, and deployment secrets. Rotate anything
  that may have been exposed in history; deleting the current file alone is insufficient.
- Add a launch README with the primary promise, current product screenshot/demo, public-repository
  limitation, BYOK model, local setup, deployment outline, contribution path, and links to the live
  product, support, privacy, and terms.
- Choose and add an owner-approved software license before using “open source” in product or launch
  copy. A publicly readable repository without a license is described as “public source,” not open
  source.
- Set the GitHub repository description, homepage URL, topics, and social preview to match the new
  positioning.
- Add visible source/star actions on the homepage, demo, and application About/Support context.
- Fetch the public star count on the server with at least one hour of caching.
- Failure to fetch a count must render **Star on GitHub** without a number and must never block a
  page response.
- Track outbound GitHub actions, not inferred star completions.

### R5. Analytics and privacy

- Add Umami v3 through environment-supplied script URL and website ID; production launch is blocked
  until the deployment is configured.
- Centralize explicit events behind a typed `track()` wrapper so UI components do not depend on
  the vendor global.
- Disable automatic event capture. Track normalized path pageviews and named product events only.
- Do not call `identify()` and do not send user IDs, email, GitHub identity, repository URLs/names,
  project/team IDs, prompts, model responses, API keys, errors containing provider payloads, or
  query strings.
- Track these events:
  - `demo_opened`
  - `demo_node_opened`
  - `demo_question_opened`
  - `repository_intent_submitted`
  - `github_auth_started`
  - `github_auth_completed`
  - `anthropic_setup_started`
  - `anthropic_setup_completed`
  - `repository_scan_started`
  - `repository_scan_succeeded`
  - `repository_scan_failed` with a fixed non-sensitive error category
  - `first_map_viewed`
  - `architecture_question_sent`
  - `alternatives_opened`
  - `repository_rescan_started`
  - `github_source_clicked`
  - `github_star_clicked`
- Update Privacy and deployment documentation with the analytics fields and provider.

### R6. Metadata and shareability

- Root title: **StackHatch — See how your codebase fits together**
- Root description: **Turn a public GitHub repository into a visual architecture map. Ask
  questions, compare alternatives, and re-scan as the code changes.**
- Add canonical, Open Graph, and social-card metadata using a current StackHatch self-map image.
- Add SoftwareApplication structured data using only verifiable fields.
- Give `/demo` its own title, description, canonical URL, and share image.
- Add a recognizable favicon/app mark derived from a simple connected-node motif, not an AI sparkle.

### R7. Accessibility, responsiveness, and performance

- Meet WCAG 2.2 AA contrast and interaction requirements on public and redesigned product surfaces.
- Every map interaction has a keyboard path and a non-canvas textual equivalent.
- Preserve visible focus, 44 px minimum primary touch targets, semantic headings, form labels, and
  useful error association.
- Avoid relying on category color alone; pair color with labels/icons or line styles.
- Support 320 px mobile width without horizontal page overflow.
- The homepage's static state remains useful without JavaScript; only the interactive demo and
  repository-intent enhancement require hydration.
- Target a Lighthouse performance score of at least 90 on the deployed mobile homepage, with LCP
  at or below 2.5 seconds and no layout shift caused by the demo.

### R8. Claim fidelity prerequisite

Before the new homepage is released, improve the repository evidence supplied to architecture
generation or narrow the promise. The preferred launch path is to:

- Read a bounded repository tree and README in addition to current metadata/manifests.
- Include relevant path evidence in generated node details where available.
- Record the analyzed default-branch commit SHA and scan time.
- Define size/time budgets and a clear partial-analysis state for large repositories.
- Keep all AI output editable and visibly identified as generated.
- Add evaluation fixtures using public repositories with known architectures and manually score
  main-component coverage, unsupported components, and incorrect connections.

The public site must not ship the primary “codebase fits together” claim if analysis remains limited
to root manifests without describing that limitation.

## Implementation Approach

### Content and components

- Replace the current `src/app/page.tsx` workflow/use-case arrays with content organized around the
  single repository workflow.
- Add reusable `PublicHeader`, `RepositoryIntentForm`, `PublicArchitectureDemo`, `GitHubLink`, and
  compact BYOK explanation components.

### Intent continuity

- Add a shared parser/normalizer for GitHub repository references and reuse it in the landing page,
  dashboard, project creation, and scan route.
- Carry a public repository intent through a strictly validated internal callback parameter or a
  short-lived same-origin session value.
- Extend the dashboard and focused Settings flow to resume the intent after authentication/key

- Ensure callback validation continues to reject external and protocol-relative redirects.

### Design system

- Update `src/app/layout.tsx` font imports and metadata.
- Replace generic/unused hero and hover utilities in `src/app/globals.css` with explicit public

- Keep semantic CSS custom properties as the single source of truth. Do not scatter raw brand
  colors through components.
- Validate every existing architecture category in light and dark themes after the surface change.

### Analytics

- Add an env-gated analytics script in the root layout and a typed, no-op-safe wrapper in
  `src/lib/analytics.ts`.
- Emit server-confirmed success/failure events only after the corresponding API action resolves.
- Normalize failure reasons to an allowlist such as `invalid_url`, `not_found_or_private`,

  `unknown`.
- Document the required Umami deployment configuration in `.env.example` and launch operations
  documentation.

### Suggested delivery slices

1. Claim-fidelity improvements and evaluation fixtures.
2. Shared repository-intent parsing and activation continuity.
3. Visual tokens, public shell, homepage content, and self-map poster.
4. Interactive `/demo` and preset question/alternative states.
5. Login, BYOK, dashboard, project creation, workspace, support, and remaining surface alignment.

## Edge Cases and Risks

- A repository becomes private or is deleted between intent and scan: retain the project only if it
  was explicitly created, show `not_found_or_private`, and offer a different public repository.
- GitHub rate limits the anonymous analyzer: preserve the intent, explain that the map was not
  created, and allow retry; do not blame the user's Anthropic key.
- Anthropic key is missing/invalid/rate-limited: retain the repository intent and route to the
  exact corrective action.
- A map is empty, partial, or malformed: do not present it as a successful first map; show a
  recoverable state and emit the fixed analytics error category.
- Re-scan currently replaces architecture/chat state: the confirmation must state exactly what is
  lost. If implementation changes to merge state, test locked and manually edited nodes explicitly.
- The StackHatch demo snapshot becomes stale: provenance remains visible, and release checks compare
  its commit to the current release so the team can intentionally refresh it.
- GitHub star count fetch fails or reaches a rate limit: render a count-free link.
- Analytics is blocked: all product behavior continues normally.
- JavaScript is disabled or the interactive chunk fails: the poster, textual walkthrough, and main
  repository action remain available.
- Long repository names and translated browser UI must not cause navigation or form overflow.
- Dark-mode map edges and node states need explicit visual regression coverage.

## Rollout

1. Complete the repository security/history review, public-readiness documentation, and
   owner-approved license decision; then make the repository accessible to unauthenticated
   visitors.
2. Complete the claim-fidelity prerequisite and verify the StackHatch repository map manually.
3. Configure Umami in staging with synthetic data and confirm that no repository, user, project,
   prompt, or query-string data is recorded.
4. Release the public shell, homepage, login/setup continuity, dashboard hierarchy, and demo to
   staging.
5. Run the full automated suite, production build, responsive visual review, keyboard/screen-reader
   pass, and mobile Lighthouse check.
6. Refresh the self-map against the release commit and generate the Open Graph image.
7. Deploy all aligned copy surfaces together so visitors do not cross from new positioning into an
   old generic-AI onboarding flow.
8. Verify production analytics, OAuth callback continuity, Anthropic setup, first map, GitHub links,
   support email, privacy, terms, and star-count fallback.
9. Launch first through the public GitHub repository, Show HN, relevant developer/open-source
   communities, and Product Hunt using the same plain promise and the `/demo` link.
10. Review failures and funnel exits daily during the first week. Prefer copy/error/onboarding fixes
    over adding more homepage sections.

Rollback is a normal application rollback to the previous release. The demo snapshot and analytics
integration contain no migrated user data. Repository-intent parameters must remain backward-safe
if a user returns during rollback.
